# Input Data Map

> One-row-per-entity inventory of **create / edit UI surfaces** across the app, plus a focused **gaps section** listing entities that have no UI today and the 14 `Property` fields the `/add-property` wizard intentionally defers to other routes.

## TL;DR

- **25 entities total** (`Property` is captured by `/add-property`; this file covers the remaining 24 + `PropertyComparable` which is not built yet).
- **5 entities are `built`** with real create or edit UI: Document, Folder, Successor, SuccessorPropertyAssignment, UserProfile, NotificationPreference (NotificationPreference and UserProfile are edit-only — records are auto-created on signup / seeded).
- **3 entities are `stub`** — visible button or affordance, no working handler: CoOwner, EmergencyContact, Professional.
- **13 entities are `seed-only`** — backend layer + (sometimes) server action exists, but no UI input path; demo data is the only source: Tenant, Lease, Payment, Expense, MaintenanceItem, Inspection, Certification, SafetyRisk, LandParcel, PropertyValuation, OwnershipRecord, OwnershipDocument.
- **3 entities are `derived`** — no UI by design (auto-generated by events): OwnershipHistory, EstateActivityEvent, Notification.
- **1 entity is `not-built`** — referenced as a gap but no schema yet: PropertyComparable.
- **14 `Property` fields are DEFERRED-BY-DESIGN** by the wizard — listed in the Gaps section with target-route assignments (mirrors `pages/add-property/audit.md` § PF10).

## Glossary

- **`built`** — Real working create or edit form ships in the UI. User can persist a new record (or edit an existing one) through normal use.
- **`stub`** — Visible affordance exists (button, link, icon) but it doesn't lead to a working form. Typically: button has no `onClick`, opens a modal that doesn't exist, or modal exists but `onSubmit` is unwired.
- **`seed-only`** — No UI input path. DB layer (`lib/data/db/<entity>.ts`) may have `create()` and a server action may exist, but no component calls it. The only records in the system come from JSON files under `public/data/users/demo-user/<entity>/`.
- **`derived`** — No UI by design. Records are auto-generated by another action (e.g. every property write logs an `OwnershipHistory` entry; every successor assignment emits an `EstateActivityEvent`).
- **`not-built`** — Entity referenced in code or audit corpus but no Zod type, no DB layer, no seed data yet.

## Gaps — places to put input UI

> A focused, prioritized list of entities and fields that need input UI today. The list combines (a) `seed-only` and `stub` entities from the inventory above, (b) the `not-built` entity, and (c) the 14 `Property` fields the `/add-property` wizard defers to other routes per `pages/add-property/audit.md` § PF10.

**Priority legend:** **P1** — blocks pages from being usable with real data (audit corpus already flagged the gap). **P2** — needed for full feature parity. **P3** — nice-to-have.

## Build order — recommended sequence

When converting gaps into wiring work, address them in this order:

1. **Fix the 2 P0 bugs** in `pages/add-property/audit.md` (PF1 lat/lng swap, PF2 Step 4 file inputs) — before adding any new input UI, the existing one needs to work.
2. **P1 wizard-internal gaps** — PF9 (`status` UI field), PF3 (`buyNumeric` from `purchasePrice` or `currentMarketValue`).
3. **P1 entity-level gaps in `/property/[id]/rental`** — Tenant + Lease (one modal), Payment, Expense, MaintenanceItem (wire existing button). Unlocks the rental dashboard for real-data use.
4. **P1 ownership** — CoOwner (wire existing button), `purchasePrice` input on ownership tab.
5. **P1 PropertyComparable** — Phase 6.9 entity build, then comparable surfaces unlock on valuation + location pages.
6. **P2 batch** — Inspection / Certification / SafetyRisk / EmergencyContact (4 wires on safety tab, currently deferred per user), PropertyValuation "Add Snapshot", OwnershipRecord / OwnershipDocument edit affordances, Professional add modal, LandParcel edit panel, remaining `Property` field inputs on overview specs section.
7. **P3** — bedrooms / bathrooms / parking / storage Property fields on overview specs.

Each item becomes a separate phase in `phases/PHASES.md` once scheduled.
