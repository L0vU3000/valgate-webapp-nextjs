# Plan — Phase 3: Create master `pages/INDEX.md` (cross-page entity roll-up)

## Context

After running 7 page audits (Phase 4), there will be 7 separate Entity Backlog sections — one inside each page's `plan.md`. Each one answers "what entities does THIS page need?" but none of them answer the actual decision-driving question:

> "If I build entity X, how many surfaces **across the whole app** does it unlock, and which entity should I build first?"

That cross-page roll-up has no home today. Without it, every prioritization decision requires opening 7 plan.md files and tallying entity mentions in your head. Phase 3 builds that one document.

The intended outcome: after Phase 3, anyone who wants to know "what should we build next?" opens **one file** — `.claude/data-audit/pages/INDEX.md` — and sees a ranked table answering the question. The detail still lives in each page's `plan.md`; this file is purely a roll-up + ranking.

## Prerequisites

- **Phase 1 complete** — `audit-page-datapoints/SKILL.md` describes the folder + audit/plan output structure AND includes the regeneration step ("after writing audit.md and plan.md, rewrite `pages/INDEX.md` from all existing plan.md files"). Without this, the file would go stale immediately.
- **Phase 2 complete** — `pages/property-id-overview/plan.md` exists and contains the 6-entity backlog that seeds this file. Without it, Phase 3 has no real data and would just be an empty template.

## Scope of this change

**One file created:** `.claude/data-audit/pages/INDEX.md`.

**No other files touched.** This is purely additive. The existing `.claude/data-audit/INDEX.md` (per-datapoint + page-level audits) keeps its current role; the new `pages/INDEX.md` is a **different** kind of index — entity-centric, not audit-centric.

The two index files answer different questions:

| File | Question it answers |
|---|---|
| `.claude/data-audit/INDEX.md` | "What audits have been written, and what did they conclude?" |
| `.claude/data-audit/pages/INDEX.md` | "Which entity unlocks the most surfaces across all pages? What should I build first?" |

## File format

```markdown
# Cross-page Entity Backlog

> Master roll-up of entity needs across all page audits. Auto-regenerated by `/audit-page-datapoints` on every run — do **not** edit manually. To change a row, edit the corresponding page's `plan.md` Entity Backlog section and re-run the skill.
>
> **Sort order:** by `Total surfaces unlocked` descending. Ties broken alphabetically by entity name.
> **Last regenerated:** `<date>` (after `pages/<slug>/plan.md` was written/updated).

## Ranked entities

| Rank | Entity | Status | Surfaces unlocked | Pages it touches | Catalog ref |
|---|---|---|---|---|---|
| 1 | Lease + Tenant | not built | 4 | [property-id-overview](property-id-overview/plan.md) (rows 8, 13, 14, 16) | [ref/00 §4 + §5](../ref/00-entity-catalog.md) |
| 2 | Payment + Expense | not built | 3 | [property-id-overview](property-id-overview/plan.md) (rows 10, 11, 12) | [ref/00 §6](../ref/00-entity-catalog.md) |
| 3 | PropertyValuation | not built | 1 | [property-id-overview](property-id-overview/plan.md) (row 7) | [ref/00 §16](../ref/00-entity-catalog.md) |
| 4 | RentalEvent / ActivityEvent | not built | 1 | [property-id-overview](property-id-overview/plan.md) (row 15) | [ref/00 §8](../ref/00-entity-catalog.md) |
| 5 | Notification + MaintenanceItem | not built | 1 | [property-id-overview](property-id-overview/plan.md) (row 16) | [ref/00 §7 + §9](../ref/00-entity-catalog.md) |

## Status legend

- **not built** — entity not yet defined in `lib/data/types/`. Wiring is blocked.
- **in progress** — entity PR open / in review.
- **shipped, partial wiring** — entity exists but some consumer pages haven't been wired yet.
- **shipped, fully wired** — entity exists and every consumer page identified by page audits is wired. Entity is "done" from the cross-page perspective.

## Per-page contribution detail

> Click any page slug in the table above to jump to that page's `plan.md` Entity Backlog section, which has the full per-row reasoning, audit batching strategy, and notes.

## How this file is regenerated

Every time `/audit-page-datapoints` runs (whether on a new route or as a re-audit on an existing one), the skill:

1. Reads every `plan.md` under `.claude/data-audit/pages/*/plan.md`
2. Aggregates each plan's `## 3. Entity Backlog` section — entities, surfaces unlocked per page, statuses
3. Sums `Surfaces unlocked` across pages per entity
4. Re-sorts by total surfaces unlocked (desc)
5. Rewrites this file in place

This means the file is **always derived** from the per-page plans. If you want to fix or refine an entity row, edit the corresponding `plan.md` Entity Backlog section, then re-run `/audit-page-datapoints` against that route. Editing `pages/INDEX.md` directly will be overwritten on next run.

## Initial seed (this Phase only)

For the initial write, use the data already produced for `pages/property-id-overview/plan.md` (post-Phase-2 location). The seed table will have **5 rows** (matching the 6-entity backlog from the migrated overview audit, with Lease + Tenant treated as one entity per the existing plan's "Land first" guidance). Subsequent Phase 4 audits will add new rows and may shift counts (e.g. Payment + Expense's surface count will jump if rental tab also needs them).
```
