# Phase 3 — Cross-Request Cache with unstable_cache + Tags

**Part of:** the 4-phase client performance plan. Phase 1 (DB filtering) and Phase 2 (React `cache()` per-request dedup) are shipped. This is Phase 3. Phase 4 (Redis) stays deferred until measured.

## Objective

Phase 1 made each query **smaller**. Phase 2 stopped the app from running the **same** query multiple times **inside one page load**. Phase 3 stops re-fetching unchanged data **across page loads and navigations** until something actually mutates.

Open `/property/PROP-0001/overview`, click Rental, click back to Overview — today every tab change re-runs every `list*` Drizzle query against Neon even though nothing changed. `unstable_cache` from `next/cache` stores the result **server-side** keyed by org + entity + optional property filter, and the **write path already emits invalidation tags** via `revalidateFeTag()` in `app/actions/*.ts`. The read path just never consumed them — Phase 3 connects the two halves.

<Callout id="decisions-locked" tone="decision">

**DECISIONS LOCKED**

1. **Mechanism = `unstable_cache` only.** No Redis (Phase 4), no blanket `"use cache"` route segments in the first cut. Next 15.5.15 already ships `unstable_cache` + `revalidateTag`.
2. **Scope = property-tab reads first.** Wire cached wrappers for the entity lists property loaders call (`overview`, `rental`, `financials`, …). Portfolio, analytics, Pro cockpit, and `getProgressContext()` are a fast-follow — not bundled into cut 1.
3. **Auth stays per-request.** `requireCtx()` keeps React `cache()` only — **never** `unstable_cache`. Clerk `auth()` must run every request; only org-scoped **data reads** are cross-request cached.
4. **Tag contract = existing flat tags.** Reads and writes both use the strings already in `app/actions/*` (`"leases"`, `"properties"`, `"documents"`, …). Cache **keys** include `orgId` (+ `propertyId` when filtered) so entries never cross tenants; flat tags mean coarse invalidation (any org's lease write busts all orgs' lease cache — acceptable v1 tradeoff, no action-file churn).
5. **Split ship.** (a) `lib/data/cached-reads.ts` helper + `getProperties` / `getPropertyByIdParam` cross-request layer, prove hit on second navigation; (b) swap property-tab `queries.ts` list calls to cached wrappers; (c) fast-follow portfolio + `property/actions.ts` tag alignment.
6. **Do not wrap `list*` services in React `cache()`.** Phase 2 locked that deferral; Phase 3 uses `unstable_cache` at the data-helper layer instead.

</Callout>

## Core new module — `lib/data/cached-reads.ts`

One helper per entity the property tabs load. Each wraps the existing `list*` service (Phase 1 filtered signatures unchanged). Primitive cache keys only — **never** a `Ctx` object reference.

<Code id="cached-reads-stub" filename="lib/data/cached-reads.ts" language="ts" code={"import \"server-only\";\nimport { unstable_cache } from \"next/cache\";\nimport { listLeases } from \"@/lib/services/leases\";\nimport type { Ctx } from \"@/lib/services/_mapping\";\nimport type { Lease } from \"@/lib/data/types/lease\";\n\n/** Cross-request cached leases for one org, optionally filtered to one property. ## Why it's safe

- **No cross-user leak:** `unstable_cache` keys always include `ctx.orgId` (and `propertyId` when scoped). Tags are global but entries are keyed per org — org A cannot read org B's cached rows.
- **Auth every request:** `requireCtx()` is not cached cross-request; only data fetches after auth succeed.
- **Stale-after-write:** every shipped mutation action already calls `revalidateFeTag` for its entity — once reads use matching tags, writes bust cache automatically.
- **Composes with Phase 2:** React `cache()` dedupes within a request; `unstable_cache` dedupes across requests. Same function names at call sites for loaders; only import path changes from `listX` → `cachedListX`.
- **Notifications exception preserved:** cache the org-wide `listNotifications` result; keep `notificationMatchesProperty()` JS filter in the loader (Phase 1 locked exception).
