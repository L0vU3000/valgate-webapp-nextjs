# AI Overlay — Artifacts Reference

> Product reference for what the AI agent overlay should surface as **artifacts** in chat and under **Workspace Assets**.  
> Use this before planning implementation logic, UI, or storage.  
> **Status:** reference only — not implemented.

---

## Purpose

Artifacts are durable outputs tied to an AI session: things the owner can revisit, open, or export after a conversation. They should reflect what owners actually do in Valgate — not generic chat attachments.

Today the overlay already links existing documents via `artifactDocIds` on messages and lists portfolio documents/charts in Workspace Assets. This doc defines the fuller catalog and how to grow it in phases.

---

## Artifact catalog (Valgate-fit candidates)

Grouped by what owners do in the app. These are **good candidates to standardize** — each maps to real entities, derivations, or export flows elsewhere in the product.

### Documents & legal

| Artifact | What it contains |
|----------|------------------|
| **Lease summary** | Key dates, rent, tenant, renewal window |
| **Ownership / title brief** | Who owns what %, linked deeds |
| **Inspection or safety report** | Findings + open items |
| **Estate assignment memo** | Successor + share per property |

### Financial

| Artifact | What it contains |
|----------|------------------|
| **Yield / cash-flow snapshot** | One property or portfolio (same intent as the Yield tool in Workspace Assets) |
| **Rent collection report** | Expected vs collected for a month |
| **Expense breakdown** | By property or by category |

### Charts & visuals

| Artifact | What it contains |
|----------|------------------|
| **Portfolio value trend** | Bar chart (like the trend card in the assets panel) |
| **Occupancy over time** | Time series |
| **Rent vs expenses comparison** | Comparative chart |

### Action / planning

| Artifact | What it contains |
|----------|------------------|
| **Progress checklist** | What’s missing to improve property health score |
| **“Next steps” brief** | e.g. upload lease, add certification |
| **Comparison table** | Two properties side by side |

### Exports (downloadable)

| Artifact | What it contains |
|----------|------------------|
| **PDF summary** | Shareable brief for accountant or lawyer |
| **CSV export** | Payments list or documents list |

## Phased rollout (recommendation)

### Phase 1 — References (cheap, matches today)

- Keep linking **existing entities**: documents, valuations, leases.
- No new files generated by the agent.
- Storage: `artifactDocIds` on the message + a **type label** (or `kind: "document-ref"`).
- UI: chips/links in chat; same items appear in Workspace Assets filters (Documents, Legal, etc.).

### Phase 2 — Generated snapshots (medium)

- Agent creates **read-only views** stored per session:
  - Markdown report body, and/or
  - Small chart config JSON (metric, property IDs, range).
- Shown inline in chat and listed under **Workspace Assets** (Reports, Charts).
- Not necessarily downloadable yet; regeneratable from session + source data if needed.

### Phase 3 — Exportable files (heavier)

- Real **PDF / CSV** generation and persistence.
- Storage path (target): `public/data/users/<userId>/ai-artifacts/` (or equivalent when backend migrates).
- `kind: "export"` with `storageId` + `mimeType`; download from overlay and assets panel.

## Target schema (aim)

Single artifact entity per session (or per message, TBD in design). Kinds align with phases above.

```ts
type AiArtifact = {
  id: string;
  sessionId: string;
  kind: "document-ref" | "report" | "chart" | "export";
  title: string;

  // document-ref → existing Document.id
  documentId?: string;

  // report → markdown body (Phase 2)
  markdown?: string;

  // chart → config for Workspace Assets / inline render (Phase 2)
  chart?: {
    metric: string;
    propertyIds: string[];
    range?: { from: string; to: string };
  };

  // export → generated file (Phase 3)
  storageId?: string;
  mimeType?: string;
};
```

### Kind → payload

| `kind` | Phase | Primary payload |
|--------|-------|-----------------|
| `document-ref` | 1 | `documentId` |
| `report` | 2 | `markdown` |
| `chart` | 2 | `chart` |
| `export` | 3 | `storageId`, `mimeType` |

## Relationship to current code (baseline)

| Area | Today |
|------|--------|
| Messages | `AiMessage.artifactDocIds?: string[]` — optional links to existing documents |
| Replies | `ai-replies` derivation can set `artifactDocIds` when a relevant doc exists |
| Chat UI | `AIMessageBubble` renders artifact links from `artifactDocIds` |
| Workspace Assets | `AIWorkspaceAssets` — portfolio documents, value trend bars, Yield link; filters: All, Documents, Charts, Legal |
| Sessions | `AiSession` — no artifact collection yet; artifacts implied per message or future session-level list |

Phase 1 extends this model (typed refs + labels). Phase 2–3 add `AiArtifact` storage and UI slots for reports, charts, and exports.

## Open questions (for a later design doc)

- **Attachment point:** artifact on message vs session-level registry vs both.
- **Id generation:** shared `idSchema` prefix (e.g. `AIA-####`) and seed data for demo-user.
- **Agent authority:** which kinds the model may create vs only suggest (especially exports).
- **Revalidation:** when source property/lease data changes, mark snapshots stale or auto-refresh.
- **Workspace Assets filters:** map `kind` + domain (Legal, Financial, Charts) to existing filter chips.
